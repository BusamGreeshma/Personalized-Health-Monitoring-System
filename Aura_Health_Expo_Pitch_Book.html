"Think about the average patient today. They track their heart rate on a smartwatch, get diagnostic blood reports as confusing PDF papers from labs, and receive prescription times written on a piece of paper. "By integrating the high-speed Llama-3 model via Groq Cloud, Aura Health dynamically aggregates biometrics, analyzes complex medical report images using vision OCR, computes chronic risk percentages, and runs a custom-engineered, local Retrieval-Augmented Generation (RAG) wellness advisory engine. Let us show you how a patient uses it in real-time."

  • What to Say: "Finally, the patient can export a complete clinical summary report. Our stylesheet is configured with native print media rules, allowing the user to print or save the document as a clean, clinical-type PDF to share with their doctor."

3. Technical Panel Q&A Cheat Sheet

Be prepared to answer these highly technical questions. These answers are designed to showcase maximum technical depth.

System Architecture
Q1: Explain the complete request-response flow when a user uploads a medical report.
"When a user uploads an image, the React frontend wraps the binary file in a `FormData` object and transmits it via `Axios` as a `POST` request to our Express endpoint. The route triggers the `uploadMiddleware` (built using `multer`) which handles file validation and saves the image temporarily. The controller then forwards the file stream to the **Groq SDK**, prompting the **`llama-3.2-11b-vision-preview`** model. We instruct the model to parse the biomarkers and return a structured JSON response. Our custom regex helper cleans conversational prefix tags out of the response, validates the JSON keys, updates the user's profile documents in MongoDB, and sends the structured telemetry payload back to the frontend to update the state."
AI & Data Extraction
Q2: LLMs are notoriously conversational. How did you ensure the backend parses Llama's output as JSON without crashing the server?
"We engineered a custom regex parser helper called `cleanAndParseJSON`. Since Llama often wraps JSON payloads in backticks or conversational explanations, our parser uses regex checks to locate the index of the first bracket `{` or `[` and the last bracket `}` or `]`. It slices the string between these indices, isolating the pure JSON string and throwing away any conversational prefixes or suffixes. This ensures that even if Llama adds conversational text, `JSON.parse()` executes flawlessly on a clean structure."
Retrieval-Augmented Generation
Q3: You claim to use RAG. Where are the embeddings stored, and how does the vector search work?
"To avoid the hosting latency and setup complexity of external cloud vector databases like Pinecone, we built a **local vector search engine in Node.js**. During setup, health articles are processed through a script (`seedKnowledgeBase.js`) where their text chunks are mapped to 768-dimensional float embedding arrays (`text-embedding-004`) and stored directly in a MongoDB collection. When a user asks a question, we generate a query vector and calculate the **Cosine Similarity** (dot product divided by the product of magnitudes) against every article vector directly in-memory using Node math functions. The top matching documents are retrieved and appended to the Llama prompt context, generating a highly targeted clinical recommendation."
Database & Modeling
Q4: Why did you choose MongoDB (NoSQL) over a SQL database like MySQL or PostgreSQL?
"Personalized health monitoring deals with highly polymorphic and semi-structured data. For example, a single daily log document contains nested structures for hydration, sleep quality, blood pressure (systolic/diastolic), and activity ring variables. In SQL, this would require complex multi-table joins or relational schemas. In MongoDB, we can store each day's unified wellness profile as a single, indexed document. This flexibility makes adding new metrics (like blood sugar or mood logs) trivial, while Mongoose schemas enforce validation constraints."
Database Performance
Q5: As your health logs grow to millions of rows, dashboard queries will slow down. How did you optimize this?
"We placed a **compound index** on the `{ userId: 1, date: -1 }` keys within the `HealthLogs` schema. Since the dashboard query looks for the logged vitals of a specific user sorted by date (newest first), this compound index allows MongoDB to locate the target document in under **10ms** by performing an index scan, completely avoiding expensive full-collection collscans."
Web APIs & Audio
Q6: How did you implement the looping medication alarm? How do you prevent it from being blocked by browser autoplay policies?
"The medication alarm uses the browser's native **Web Audio API**. We create a primary `AudioContext` and attach a gain control module and frequency oscillators. To bypass the browser autoplay policy (which blocks audio contexts from playing until a user interacts), we set event listeners for clicks and keypresses on the window layout. Upon the first user click anywhere, the `AudioContext.resume()` promise triggers, unlocking browser audio authorization. The background scheduler keeps checking times, and if a medication is due and not marked taken, it loops a dual-frequency chime (`E5` and `A5`) every 3 seconds. The moment the user clicks 'Taken', the state updates, the loop interval is cleared, and the audio context stops playing."
Security & Authorization
Q7: Explain your authentication flow. How do you secure endpoints?
"We use a standard stateless token-based flow. When a user logs in, their password query is checked using `bcryptjs.compare` against the salted hash stored in MongoDB. Upon verification, the server signs a **JSON Web Token (JWT)** containing the user's database ID and exports it. The React client stores this token in state and caches it in `localStorage`. Subsequent API requests attach this token inside the HTTP `Authorization` header (`Bearer `). The Express backend runs an `authMiddleware` that decodes the token using our `JWT_SECRET`, checks if the user exists, and appends the user context to the request object (`req.user`), rejecting unauthenticated requests with `401 Unauthorized`."
Serverless Deployment
Q8: How is the application deployed, and how does Vercel host an Express app?
"The application is split into two packages. The static React client is compiled using Vite and deployed to Vercel's Edge CDN. The Express API server is deployed using a custom `vercel.json` routing configuration. Vercel maps all incoming backend routes `/api/*` to the `server.js` script. It wraps our Express server in **Vercel Serverless Functions (@vercel/node)**. When a request hits the backend URL, Vercel boots a stateless container, executes the Express route handler, and immediately suspends the container once the response is sent. This provides instant, automated scaling without managing active servers."

4. The "WOW" Factors (Highlight These!)

To win the expo, you must highlight custom-engineered code that goes beyond a standard CRUD app. Mention these points:

  1. No Standalone Vector DB: Point out that you coded your own Vector Cosine Similarity algorithm in pure JavaScript rather than using Pinecone or other cloud database APIs. This shows you understand vector mathematics (dot products, vector magnitude, and angles).
  2. Browser-Native DSP (Digital Signal Processing): Highlight that the emergency sirens and medication remind rings are generated programmatically using sound wave formulas in the browser via `OscillatorNode` rather than downloading flat MP3 audio files. This keeps the project lightweight and network-independent.
  3. Pure CSS Responsive Glassmorphism: Show off the custom HSL design variables and frosted-glass panel effects, emphasizing that you avoided bulky CSS frameworks and tailored the theme borders and shadows yourself.
